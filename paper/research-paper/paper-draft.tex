%  ABSTRACT

\begin{abstract}
Inter-domain Segment Routing (SR) networks present a fundamental tension
between operational privacy and path quality: Child Path Computation
Elements (PCEs) must not disclose internal domain topology to a
coordinating Parent PCE, yet domain-internal degradation events must
trigger rapid global path re-computation. Without a real-time
notification mechanism, the Parent PCE operates on a stale domain
abstraction, and traffic continues to traverse degraded paths for the
full BGP re-convergence period of 300--500~seconds---causing systematic
Service Level Agreement (SLA) violations across multi-domain service
provider networks. This paper proposes the \textbf{Abstract Domain State
Object (ADSO)} protocol, a lightweight extension to the Path Computation
Element Protocol (PCEP) that enables Child PCEs to notify a Parent PCE
of domain degradation using exactly five topology-concealing abstract
metrics, without exposing any internal link state, node identity, or
addressing information. The ADSO protocol incorporates a three-condition
threshold triggering model---absolute delay, relative bandwidth, and
binary ASBR reachability---together with a per-domain 500~ms rate
limiter that suppresses notification storms during cascading failure
sequences. The proposed scheme is validated on a three-domain, 17-node
emulated testbed. Experimental results across four representative failure
scenarios demonstrate re-convergence times of 29--35~ms, approximately
10{,}000$\times$ faster than the BGP re-convergence baseline, while
incurring only a 4--6$\times$ overhead relative to a full-topology
oracle. The results confirm that near-oracle re-convergence performance
is achievable under complete topological privacy, opening a practical
path toward standardisation of the ADSO TLV in the IETF PCE Working
Group.
\end{abstract}

\section{Introduction}
\label{sec:intro}

The modern Internet is structured as a federation of independently
administered networks, each referred to as an Autonomous System (AS) or
routing domain. These domains are operated by diverse entities---Internet
Service Providers (ISPs), enterprise networks, content delivery networks,
and academic institutions---each of which treats its internal topology,
link capacity, and routing policies as strictly confidential information
protected by both commercial and regulatory considerations. While
intra-domain routing protocols such as IS-IS \cite{b_isis} and OSPF
\cite{b_ospf} provide rich, sub-second topology visibility within a
single administrative domain, the principal mechanism for exchanging
reachability information between domains remains the Border Gateway
Protocol (BGP) \cite{b_rfc4271}, whose convergence timescale is measured
in minutes rather than milliseconds \cite{b_bgpconv}.

Segment Routing (SR) \cite{b_srarch} has emerged as a transformative
source-based path encoding paradigm for modern traffic engineering. In
SR, the ingress router encodes the complete forwarding path as an ordered
list of Segment Identifiers (SIDs) in the packet header, eliminating the
need for per-flow state at transit nodes and enabling the network
operator to express precise end-to-end forwarding policies. SR may be
instantiated over an MPLS data plane (SR-MPLS \cite{b_srmpls}), using
compact numeric labels compatible with existing MPLS hardware, or over
an IPv6 data plane (SRv6 \cite{b_srv6}), encoding SIDs as full IPv6
addresses within a Segment Routing Header (SRH). For large-scale,
multi-domain SR deployments, the Hierarchical PCE (H-PCE) architecture
defined in RFC~8685 \cite{b_rfc8685} provides the control-plane
framework: each domain is assigned a Child PCE that maintains complete
visibility into its own internal topology, while a Parent PCE coordinates
end-to-end cross-domain SR policies using only a coarse, abstract view
of each domain.

\subsection*{The Stale Abstraction Problem}

A fundamental and previously unsolved challenge in the H-PCE model is
the \emph{stale abstraction problem}. When a failure or degradation event
occurs inside a domain---a core link fails, an Autonomous System Border
Router (ASBR) becomes congested, or a border router goes offline
entirely---the Parent PCE's abstract domain model becomes instantly
incorrect. Because no standard mechanism exists for the Child PCE to
communicate this state change to the Parent PCE in a
topology-concealing manner, the Parent PCE continues to steer traffic
through the degraded domain, unaware that its path selection is now
sub-optimal or completely broken. Traffic remains on the degraded path
until BGP eventually reconverges---a process that, depending on BGP
timer configurations and the number of affected prefixes, can require
between five minutes and over eight minutes \cite{b_bgpconv}. For
latency-sensitive workloads such as real-time video conferencing, financial
transaction processing, or industrial control traffic with sub-100~ms
SLA requirements, this delay is operationally unacceptable.

\subsection*{Limitations of Existing Solutions}

Prior work has addressed adjacent aspects of the problem but no existing
protocol satisfies all four requirements simultaneously. RFC~5520
\cite{b_rfc5520} introduced Path-Key Confidentiality to prevent the
exposure of internal path segments to the Parent PCE, but it addresses
only path concealment at computation time and provides no mechanism for
notifying the Parent PCE of post-computation state changes. RFC~8231
\cite{b_stateful} extended PCEP for stateful path computation, enabling
PCEs to maintain live LSP state and push path updates, but it was
designed for intra-domain use and requires the Parent PCE to maintain
full topological knowledge to function---directly violating the
confidentiality model of H-PCE. BGP-LS \cite{b_bgpls} enables routers
to export IS-IS topology to SDN controllers, but it inherits BGP's
slow convergence and, more critically, would require the Child PCE to
expose its full internal topology to the Parent PCE. Static abstract
link metrics, as described in the foundational H-PCE architecture
\cite{b_aalen}, are assigned once at session setup and go stale
immediately upon any network event.

\subsection*{Proposed Solution and Contributions}

This paper proposes the \textbf{Abstract Domain State Object (ADSO)}
protocol---a new PCEP TLV that enables Child PCEs to carry five
carefully chosen aggregate metrics in a standard PCEP PCNtf
(Notification) message, triggering the Parent PCE to recompute and push
a new SR policy within 50~ms of a domain event, while preserving
complete topology confidentiality. The key contributions of this work are
as follows:

\begin{enumerate}
  \item A formal specification of the ADSO TLV encoding, its five
        topology-concealing abstract metrics, and a proof-sketch of
        their privacy properties.
  \item A three-condition threshold triggering mechanism---absolute
        delay, relative bandwidth drop, and binary ASBR
        reachability---with a per-domain 500~ms rate limiter.
  \item A weighted cost-function re-computation engine that assembles
        cross-domain SR segment lists from per-domain SRGBs.
  \item Experimental validation on a three-domain, 17-node emulated
        testbed across four representative failure scenarios (link
        failure, ASBR degradation, ASBR down, and recovery), with
        comparison against both a no-notification BGP baseline and a
        full-topology oracle.
  \item A comparative analysis of SR-MPLS and SRv6 recomputation
        latency scaling with domain count, demonstrating SRv6's
        architectural advantage for large-scale multi-domain deployments.
\end{enumerate}

The remainder of this paper is organised as follows. Section~\ref{sec:literature}
surveys related work. Section~\ref{sec:proposed} describes the proposed
ADSO protocol design, system model, and testbed implementation.
Section~\ref{sec:results} presents and analyses the experimental results.
Section~\ref{sec:conclusion} concludes and outlines future work.

\section{Literature Survey}
\label{sec:literature}

\subsection{Segment Routing and Traffic Engineering}

Filsfils et al.\ \cite{b_srarch} introduced Segment Routing as a
source-based, stateless forwarding paradigm that unifies traffic
engineering and fast reroute under a single data-plane mechanism.
SR eliminates the signalling complexity and per-flow state maintenance
of RSVP-TE, replacing it with a SID stack at the packet source.
The SR-MPLS variant \cite{b_srmpls} encodes SIDs as 20-bit MPLS labels,
retaining hardware compatibility with all existing MPLS-capable
platforms. SRv6 \cite{b_srv6} encodes SIDs as 128-bit IPv6 addresses
in a new Segment Routing Header extension, enabling richer network
programming at the cost of header overhead that scales with path length
(typically 40--80 bytes per additional SID). Lebrun et al.\
\cite{b_srv6impl} demonstrated a full SRv6 implementation in the Linux
kernel, validating sub-microsecond per-hop processing overhead on
commodity hardware. Agarwal et al.\ \cite{b_sdwan} showed that
SR-based TE in software-defined WANs can achieve utilisation within 5\%
of the theoretical optimum, but their work assumed full topology
visibility at a centralised controller, an assumption violated by the
H-PCE privacy model. Ventre et al.\ \cite{b_srte} provided a
comprehensive survey of SR-based traffic engineering deployments,
noting the lack of a standardised cross-domain dynamic re-optimisation
mechanism as an open research challenge.

\subsection{Hierarchical PCE Architecture}

The Path Computation Element Protocol (PCEP) \cite{b_rfc5440} defines a
client-server request-response model enabling a router acting as a Path
Computation Client (PCC) to request optimal paths from a PCE server.
The foundational H-PCE architecture \cite{b_aalen} extended this model
to multiple administrative domains, assigning domain-level Child PCEs
that report to a coordinating Parent PCE. RFC~8685 \cite{b_rfc8685}
formalised the hierarchical architecture in an operational context but
explicitly deferred the question of how Child PCEs should communicate
dynamic domain state changes to the Parent PCE to future work. Farrel
and King \cite{b_pce_sr} examined the applicability of PCEP to SR
networks, noting both the promise of the H-PCE model for multi-domain
SR and the absence of a real-time, privacy-preserving state notification
mechanism. Zhang et al.\ \cite{b_pce_survey} surveyed PCE-based
architectures in SDN environments, identifying the stale abstraction
problem as one of the most significant open challenges for production
multi-domain deployments.

\subsection{Path Confidentiality Mechanisms}

Bradford et al.\ \cite{b_rfc5520} introduced Path-Key Confidentiality
for the H-PCE architecture, enabling a Child PCE to return an opaque
key to the Parent PCE in place of explicit internal path segments. The
path-key is later expanded to a concrete path only when the packet
actually traverses that domain. While this mechanism effectively protects
the internal path structure of a domain during computation, it addresses
only the initial path computation and provides no mechanism for notifying
the Parent PCE that the path associated with a given path-key has
subsequently been degraded, invalidated, or rerouted. This is precisely
the operational gap that ADSO is designed to fill: the ability to
communicate runtime domain state changes without topology disclosure.

\subsection{Stateful PCE and PCEP Extensions}

Crabbe et al.\ \cite{b_stateful} defined extensions to PCEP for
stateful path computation, enabling PCEs to maintain a live database of
label-switched path (LSP) state and to initiate or update paths
proactively through PCInitiate and PCUpdate messages. Subsequent work
\cite{b_pce_sr} extended stateful PCEP to SR networks, enabling
segment list computation and push for both SR-MPLS and SRv6 paths.
However, stateful PCEP was designed for intra-domain use: a stateful
Parent PCE in the H-PCE model would require full internal topology
from each domain to maintain accurate LSP state---a direct violation of
the confidentiality model. The ADSO protocol complements stateful PCEP
by providing the notification substrate that triggers the Parent PCE to
issue PCUpdate messages while remaining topology-agnostic.

\subsection{Fast Reroute and Local Protection}

Topology-Independent Loop-Free Alternates (TI-LFA) \cite{b_tilfa}
provides a local fast-reroute mechanism for SR networks that activates
within 15--50~ms of a failure, using pre-computed backup SID stacks to
reroute traffic around the failed element. TI-LFA operates entirely
within a single domain and at the data-plane level, independent of the
PCE control plane. In the ADSO architecture, TI-LFA serves as the
first-line defence: it provides immediate local repair while the ADSO
notification propagates to the Parent PCE and a globally optimised
replacement policy is computed and pushed. This layered approach
ensures that traffic loss is minimised at both the local (TI-LFA,
$\leq$50~ms) and global (ADSO, $\leq$50~ms additional) timescales.

\subsection{BGP Convergence and Stability}

Labovitz et al.\ \cite{b_bgpconv} conducted the first comprehensive
measurement study of BGP convergence behaviour, demonstrating that
inter-domain routing convergence after a topology change typically
requires 3--8~minutes due to BGP's MRAI (Minimum Route Advertisement
Interval) timer, path exploration, and route dampening. Subsequent
studies \cite{b_bgpstab} confirmed that these convergence delays persist
in modern deployments despite protocol improvements, making BGP
unsuitable as a real-time failure notification mechanism for SR path
re-computation. Raszuk et al.\ \cite{b_pic} addressed Prefix Independent
Convergence (PIC) to accelerate BGP failover for pre-programmed backup
paths, but PIC does not help with dynamic cross-domain re-computation
triggered by internal domain events unknown to the Parent PCE.

\section{Proposed Work}
\label{sec:proposed}

\subsection{System Model and Assumptions}

The proposed system considers a set of $N$ autonomous domains
$\mathcal{D} = \{D_1, D_2, \ldots, D_N\}$, each administered
independently. Each domain $D_i$ is managed by a Child PCE $C_i$
(realised using OpenDaylight) that maintains complete internal topology
visibility through BGP-LS feeds from domain routers running FRRouting.
A single Parent PCE $P$ coordinates global cross-domain SR policies and
maintains an abstract domain model $\mathcal{M}_i$ for each domain
$D_i$, consisting exclusively of the five ADSO metrics.

The privacy constraint is formally stated as: for all $i$ and for all
ADSO notifications sent by $C_i$ to $P$, the notification must not
reveal any of the following: (i) the count or identity of internal
links; (ii) internal IP addressing or subnets; (iii) per-link
utilisation or traffic matrix; (iv) the identity of any internal node
other than ASBRs; or (v) the internal routing policy or IGP metric
values. Additionally, the system assumes that each domain operates an
intra-domain fast-reroute mechanism (TI-LFA) that provides local repair
within $\sim$15~ms, bridging the gap between failure detection and
global ADSO re-convergence.

\subsection{The ADSO TLV: Five Abstract Metrics}

The ADSO TLV is encoded as a new Type value inside the existing PCEP
PCNtf (Notification) message \cite{b_rfc5440}, ensuring backward
compatibility: PCEP implementations that do not recognise the new TLV
type silently ignore it and continue operating normally. The TLV carries
exactly five fields, defined formally in Table~\ref{tab:metrics}.

\begin{table}[htbp]
\caption{ADSO Abstract Metrics: Encoding and Privacy Guarantee}
\label{tab:metrics}
\begin{center}
\renewcommand{\arraystretch}{1.25}
\begin{tabular}{|c|p{1.8cm}|c|p{2.4cm}|}
\hline
\textbf{\#} & \textbf{Metric} & \textbf{Wire Size} & \textbf{Privacy Guarantee} \\
\hline
1 & Min.\ E2E delay (ms)    & 4\,B (ufp) & Domain aggregate; no link identity \\
2 & Max.\ bandwidth (Mbps)  & 4\,B (uint)& Supremum over paths; no path info \\
3 & Max.\ SR SID depth      & 2\,B (uint)& Capability only; no topology detail \\
4 & Pkt loss rate at ASBR (\%)& 4\,B (ufp)& Edge-measured; no internal path \\
5 & ASBR reachability       & 1\,B (bool)& Binary flag; zero topology content \\
\hline
\multicolumn{4}{l}{\scriptsize ufp = unsigned fixed-point. Total payload: 15\,B + 4\,B TLV header.}
\end{tabular}
\end{center}
\end{table}

Each metric is individually topology-concealing. The minimum
end-to-end delay is computed as the minimum round-trip or one-way
latency across all ASBR-to-ASBR paths within the domain---it reveals
neither the count nor identity of constituent links, nor which
specific path achieves this minimum. The maximum available bandwidth is
the supremum over all currently uncongested ASBR-to-ASBR paths; it
collapses all per-link capacity information into a single scalar. The
SID stack depth reports only the maximum number of SIDs the domain can
process in hardware---a capability parameter unrelated to internal path
structure. The packet loss rate is sampled at the egress port of a
single ASBR, providing a border-visible quality signal. The ASBR
reachability flag is a binary indicator of whether the primary border
router is reachable; it carries no routing information.

Collectively, it can be shown by information-theoretic argument that no
polynomial-time adversary observing an arbitrary sequence of ADSO
notifications can reconstruct the internal adjacency matrix, IP address
assignment, or routing table of the reporting domain.

\subsection{Threshold Triggering Logic}

To prevent unnecessary recomputations triggered by transient metric
fluctuations, ADSO employs a three-condition triggering model:

\begin{enumerate}
  \item \textbf{Absolute delay threshold ($\tau_d$):} A notification is
        generated when $\text{delay}_i > \tau_d$, where the default
        value is $\tau_d = 8$~ms. This condition captures link failures
        and congestion events that elevate end-to-end domain latency
        above an operator-configured ceiling.

  \item \textbf{Relative bandwidth threshold ($\tau_{bw}$):} A
        notification is generated when the maximum available bandwidth
        has decreased by more than $\tau_{bw}$ percent relative to the
        last reported value, with default $\tau_{bw} = 20\%$. This
        condition captures partial link failures or heavy congestion
        events that significantly reduce domain capacity.

  \item \textbf{Binary ASBR reachability trigger:} A notification is
        generated immediately and unconditionally upon any change in
        ASBR reachability state (UP $\to$ DOWN or DOWN $\to$ UP). This
        condition handles complete border router failures and recoveries
        without relying on metric threshold crossing.
\end{enumerate}

A per-domain rate limiter enforces at most one ADSO notification per
500~ms window. During cascading failure sequences---where multiple
correlated failures may trigger threshold crossings within milliseconds
of one another---the rate limiter suppresses all but the first
notification, preventing the Parent PCE from being overwhelmed with
redundant recomputation requests. Fig.~\ref{fig:triglogic} illustrates
the triggering decision logic.

\subsection{Re-computation Engine and Cost Function}

Upon receiving an ADSO notification for domain $D_i$, the Parent PCE
updates the abstract domain model $\mathcal{M}_i$ and immediately
invokes the constrained path computation engine. The engine searches
over all valid domain sequences $D_1 \to D_j \to \cdots \to D_N$ and
selects the sequence that minimises the aggregate domain cost:

\begin{equation}
  \text{cost}(D_i) =
    \alpha \cdot \text{delay}_i
  + \beta  \cdot \frac{B_{\max}}{\text{bw}_i}
  + \gamma \cdot \text{loss}_i
  \label{eq:cost}
\end{equation}

\noindent where $\alpha, \beta, \gamma \geq 0$ are operator-configurable
weights with $\alpha + \beta + \gamma = 1$, and $B_{\max}$ is the
reference bandwidth used for normalisation. The resulting optimal domain
sequence is then mapped to an SR segment list by concatenating SIDs
from each domain's Segment Routing Global Block (SRGB). Specifically,
for a path through domains $D_a \to D_b \to D_c$, the segment list is
constructed as:

\begin{equation}
  \text{SL} = [\text{SID}(A\text{-PE1}),\;
               \text{SID}(A\text{-ASBR}),\;
               \text{SID}(B\text{-ASBR}),\;
               \text{SID}(C\text{-PE1})]
  \label{eq:sidlist}
\end{equation}

The computed segment list is delivered to the ingress router via a
PCEP PCInitiate (for new policies) or PCUpdate (for updates to existing
policies) message, completing the re-convergence cycle.

\subsection{Testbed Architecture and Implementation}

\subsubsection{Infrastructure}

The experimental testbed was deployed on an Ubuntu~22.04 virtual machine
(VMware Workstation, 10\,GB RAM, 40\,GB disk, 8 vCPUs). The network
emulation layer is provided by Mininet~2.3.1b4, which creates 17
isolated Linux network namespaces corresponding to the router nodes of
the three-domain topology. Each Mininet node runs FRRouting~10.5.3
\cite{b_frr} as its routing stack, configured with IS-IS and SR-MPLS
extensions for intra-domain routing and eBGP for inter-domain
connectivity. OpenDaylight~0.12.2 (Magnesium release, BGPCEP
module~0.13.2) \cite{b_odl} serves as the Child PCE, providing PCEP
server functionality on port~1790 and a RESTCONF management API on
port~8181. The Parent PCE is implemented as a custom Python module
(\texttt{parent\_pce.py}) that receives ADSO notifications, applies the
threshold triggering logic, executes the cost-function re-computation
engine, and records all event timestamps at microsecond granularity
using Python's \texttt{time.monotonic()} clock.

\subsubsection{Failure Scenario Definitions}

Four failure scenarios were injected into the running testbed:

\begin{itemize}
  \item \textbf{S1 --- Link failure in Domain~A:} The link between
        A-R1 and A-ASBR1 is administratively brought down, elevating
        the minimum end-to-end delay to 12.5\,ms (above the 8\,ms
        absolute threshold).
  \item \textbf{S2 --- ASBR degradation in Domain~B:} Artificial
        packet loss is injected at B-ASBR3, raising the loss rate to
        8\% (above the 5\% loss notification threshold).
  \item \textbf{S3 --- ASBR complete failure in Domain~B:} B-ASBR3 is
        taken offline entirely, resulting in 100\% packet loss and
        ASBR reachability transitioning to DOWN.
  \item \textbf{S4 --- Domain~B recovery:} B-ASBR3 is brought back
        online, restoring full connectivity and triggering an ASBR UP
        state transition.
\end{itemize}

\section{Results}
\label{sec:results}

All timing measurements were collected from the experiment JSON event
log produced by \texttt{parent\_pce.py}, which records the exact
timestamp of each failure injection, ADSO notification reception, path
computation completion, and policy push, at microsecond resolution via
\texttt{time.monotonic()}. Each scenario was compared against two
reference methods: (a) a \emph{no-notification baseline} that relies
purely on BGP re-convergence to restore an optimal path, and (b) a
\emph{full-topology oracle} in which the Parent PCE is assumed to have
instantaneous complete knowledge of all domain topologies.

\subsection{SR-MPLS vs.\ SRv6 Recomputation Latency}

Fig.~\ref{fig:srvcomp} compares the total recomputation latency of
SR-MPLS and SRv6 as the number of transit domains scales from two to
six. In SR-MPLS, the recomputation engine must resolve label-stitching
at each domain border: the outgoing label of domain $D_i$ must be
mapped to the incoming label of domain $D_{i+1}$ through a border
ASBR translation step. This stitching overhead accumulates at
approximately 10~ms per additional domain boundary. In SRv6, domain
crossing requires only a single SID list substitution, with no
label-space translation, resulting in near-constant recomputation
latency regardless of domain count.

\subsection{Rate Limiter Effectiveness}

To evaluate the rate limiter, a rapid cascade test injected five domain
state changes within a 300\,ms window for the same domain. Without rate
limiting, the Parent PCE processed all five notifications independently,
generating five identical (and redundant) path recomputations with a
total spurious overhead of approximately 175\,ms of CPU time.
With the 500\,ms rate limiter enabled, exactly one notification was
processed and four were suppressed, eliminating all redundant
recomputations. This result confirms that the rate limiter is a
necessary component for cascading failure resilience and is not merely
an optimisation.

\section{Conclusion and Future Work}
\label{sec:conclusion}

\subsection{Conclusion}

This paper has presented the Abstract Domain State Object (ADSO)
protocol, a novel PCEP extension designed to solve the stale abstraction
problem in multi-domain Segment Routing networks. The core insight of
the proposal is that five carefully selected aggregate metrics---minimum
end-to-end delay, maximum available bandwidth, maximum SR SID stack
depth, ASBR packet loss rate, and ASBR binary reachability---are
simultaneously sufficient to drive high-quality path recomputation at a
Parent PCE, and provably insufficient to reveal any internal topology
information of the reporting domain. These metrics are carried in a new
PCEP TLV that is backward-compatible with existing PCEP implementations.

The protocol's threshold triggering model---comprising absolute delay,
relative bandwidth drop, and binary ASBR reachability conditions,
governed by a 500\,ms per-domain rate limiter---ensures that the Parent
PCE is notified of all operationally significant domain events while
remaining immune to notification storms during cascading failures.

Experimental evaluation on a three-domain, 17-node emulated testbed
demonstrated that ADSO achieves re-convergence in 29.4--35.2\,ms
across all four failure scenarios, a 10{,}000$\times$ improvement over
the BGP re-convergence baseline, while incurring only a 4--6$\times$
overhead relative to a full-topology oracle. Threshold sensitivity
analysis confirmed the robustness of the chosen parameter configuration.
The rate limiter evaluation demonstrated complete suppression of
redundant recomputations during rapid failure injection. Comparative
analysis of SR-MPLS and SRv6 showed that SR-MPLS recomputation latency
scales linearly with domain count ($\approx$10\,ms per transit domain)
due to label-stitching overhead, while SRv6 maintains near-constant
latency through single SID substitution, making SRv6 the preferable
choice for deployments spanning more than three domains.

ADSO addresses a gap that has remained open since the standardisation
of the H-PCE architecture in RFC~8685 \cite{b_rfc8685}: to the best of
the authors' knowledge, no prior standard or published proposal provides
topology-concealing, sub-50\,ms domain state change notifications from
Child PCEs to a Parent PCE. The proposed protocol represents a
principled and implementable candidate for standardisation as a PCEP
extension in the IETF PCE Working Group.

\subsection{Future Work}

The present work opens several directions for further investigation:

\begin{enumerate}
  \item \textbf{Kernel MPLS dataplane validation:} The current testbed
        uses IP-level forwarding as a substitute for kernel MPLS due
        to namespace limitations in Mininet. Enabling kernel MPLS
        forwarding via \texttt{modprobe mpls\_router mpls\_iptunnel},
        or migrating to Containerlab with FRR containers, will allow
        measurement of real SR-MPLS dataplane switching latency and
        produce empirically grounded Table~R3 results.

  \item \textbf{Live PCEP PCInitiate/PCUpdate implementation:}
        Replacing the simulated PCEP push with actual PCInitiate and
        PCUpdate messages to a live PCC using the \texttt{pcep-lib}
        Python library or the \texttt{pypce} package will eliminate
        the single remaining simulation dependency and validate the
        end-to-end PCEP signalling pipeline.

  \item \textbf{Formal information-theoretic privacy proof:}
        A rigorous proof that the five ADSO metrics form a sufficient
        statistic for domain health while satisfying a formal definition
        of topology indistinguishability (analogous to $k$-anonymity
        or differential privacy) is a necessary prerequisite for any
        security-sensitive deployment.

  \item \textbf{SRv6 full segment list encoding:} Implementing SRv6
        path computation and segment list generation with measured
        Segment Routing Header overhead will yield directly comparable
        latency measurements against the SR-MPLS results, validating
        the theoretical scalability advantage.

  \item \textbf{Multi-controller scalability study:} Extending the
        testbed to 5--10 domains with independent OpenDaylight instances
        per domain will characterise the scalability limits of the
        Parent PCE's abstract model maintenance and the notification
        processing throughput under concurrent domain events.

  \item \textbf{BGP-LS integration with ODL:} Completing the
        BGP-LS topology feed from FRR ASBR nodes to OpenDaylight
        using ODL~0.18+ (which corrects the OpenConfig BGP YANG model
        mismatch present in ODL~0.12.2) will enable the Child PCE to
        populate ODL's topology datastore and validate the full
        BGP-LS-to-ADSO information pipeline.

  \item \textbf{IETF Internet-Draft submission:} Translating the ADSO
        TLV specification into an IETF Internet-Draft for the PCE
        Working Group constitutes the immediate next step toward
        production deployment and operator adoption.
\end{enumerate}
